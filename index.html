01 — Introducción

Tres herramientas, un mismo motor

Desarrollado por la Open Information Security Foundation (OISF), Suricata reúne en un único proceso las capacidades de un Sistema de Detección de Intrusiones (IDS), un Sistema de Prevención de Intrusiones (IPS) y un monitor de seguridad de red (NSM).

IDS / IPS

Detecta tráfico malicioso y, en modo inline, puede bloquearlo antes de que llegue a su destino.

NSM

Registra cada flujo y evento de la red, dejando un rastro completo para investigaciones posteriores.

Inspección profunda

Decodifica protocolos en las capas 3, 4 y 7 del modelo OSI para entender el tráfico, no solo contarlo.

Arquitectura multihilo

Distribuye el análisis entre los núcleos del procesador para sostener enlaces de alto volumen.

Arquitectura

Un punto de vigilancia entre el perímetro y la red interna

Suricata se ubica entre el firewall y la red interna, inspeccionando todo el tráfico que pasa por ese punto. Lo que detecta alimenta al mismo tiempo al SIEM, para correlación automática, y al analista de seguridad, que revisa los registros directamente.

03 — Configuración

suricata.yaml: el panel de control

Todo el comportamiento del motor se ajusta desde un único archivo, ubicado en /etc/suricata/suricata.yaml.

HOME_NET

Define los rangos de IP que se consideran red interna o de confianza.

EXTERNAL_NET

Define el tráfico externo, generalmente configurado como !$HOME_NET.

af-packet

Especifica la interfaz de red que Suricata escuchará activamente.

outputs / eve-log

El formato JSON estándar para integrarse con un SIEM o con Kibana.

app-layer

Activa o desactiva la detección automática de protocolos como HTTP, DNS, TLS o SMB.

rule-files

Indica las rutas a los ficheros .rules que se cargarán al iniciar, incluyendo las de Emerging Threats.

MODO IDS

Solo lectura. Monitorea y genera alertas sin bloquear nada — ideal para tener visibilidad sin riesgo.

MODO IPS

Inline. Intercepta el tráfico y puede bloquearlo activamente — protección en tiempo real.

MODO NSM

Registro completo del tráfico, pensado para análisis forense y auditoría posterior.

04 — Reglas

Anatomía de una regla

Toca cada parte de la regla para ver qué función cumple dentro de la detección.

alert tcp $EXTERNAL_NET any -> $HOME_NET 22 ( msg:"Posible escaneo SSH"; flags:S; threshold: type both, track by_src, count 5, seconds 60; sid:1000001; rev:1; )

Esta regla genera una alerta cuando un host externo intenta 5 o más conexiones TCP al puerto 22 en menos de 60 segundos — un patrón típico de fuerza bruta o escaneo de puertos.

Cabecera

Define el tráfico al que aplica la regla: protocolo, IP y puerto de origen, dirección del flujo (->) e IP y puerto de destino.

msg

Mensaje descriptivo que aparece en la alerta generada.

content

Cadena de texto o bytes que Suricata busca dentro del payload del paquete.

sid / rev

Identificador único de la regla y número de revisión para llevar control de versiones.

priority

Nivel de criticidad de la alerta, del 1 (crítico) al 4 (informativo).

Escaneo de puertos

Detectar reconocimiento mediante reglas que identifiquen múltiples conexiones SYN a distintos puertos desde un mismo origen.

Tráfico HTTP sospechoso

Reglas que busquen cadenas maliciosas en peticiones HTTP, como patrones de inyección SQL o intentos de traversal de directorios.

Comunicación C2

Identificar conexiones salientes a infraestructura de comando y control mediante reglas de DNS y TCP.

evidencia real · vm propia (suricata 8.0.5 · ubuntu server 24.04 lts)

Instalé Suricata en una máquina virtual propia (Ubuntu Server 24.04 LTS sobre VirtualBox), escribí una regla local y la dejé corriendo en modo escucha sobre la interfaz de red. Al ejecutar ping -c 5 8.8.8.8 desde la propia VM, el motor detectó el tráfico ICMP y generó la alerta en tiempo real, registrada en /var/log/suricata/fast.log con la firma de mi regla: alert icmp any any -> any any (msg:"Ping detectado - regla local"; sid:1000001; rev:1;)

Buenas prácticas

Cuatro hábitos para mantener Suricata afilado

Mantener reglas actualizadas

Ejecutar suricata-update con regularidad para incorporar las últimas firmas de Emerging Threats.

Ajustar HOME_NET

Definir con precisión los rangos de red interna para reducir falsos positivos y mejorar la relevancia de las alertas.

Revisar rendimiento

Monitorizar el uso de CPU y memoria, y ajustar los hilos en suricata.yaml según los recursos disponibles.

Integrar con SIEM

Enviar el eve.json a plataformas como Elastic Stack o Splunk para correlación avanzada de eventos.
